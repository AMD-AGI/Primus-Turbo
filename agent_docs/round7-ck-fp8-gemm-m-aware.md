# Round 7: CK FP8 GEMM M-aware Tile Selection

## 优化概述

为 CK FP8 GEMM (非 Grouped) 添加 M 维度感知的 tile 选择。当 256-row tile 产生的 tile
数量不足以填满全部 CU 时，自动切换到 128x128x128 tile 以提升 CU 利用率。

## 修改文件

| 文件 | 变更 |
|------|------|
| `csrc/kernels/gemm/ck_gemm_kernel_instance_factory.cu` | GFX942/GFX950 添加 M-aware 分支 |
| `csrc/kernels/gemm/ck_gemm_kernel_instance_factory.hip` | 同步 hipify 版本 |

## 实现逻辑

```
对于 RowColQuant / TensorQuant 路径 (非 ABQuantGrouped):

  IF k % 128 != 0:
      → 256x128x128 padK (不变，需要 padding)
  ELSE IF n % 256 == 0:
      total_tiles = ceil(m/256) × (n/256)
      IF total_tiles < NUM_CU → 128x128x128  // M-aware 触发
      ELSE                    → 256x256x128  // 原始路径
  ELSE IF n % 128 == 0:
      total_tiles = ceil(m/256) × (n/128)
      IF total_tiles < NUM_CU → 128x128x128  // M-aware 触发
      ELSE                    → 256x128x128  // 原始路径
  ELSE:
      → 256x128x128 padding (不变)

  NUM_CU: GFX942=304 (MI300X), GFX950=256 (MI355X)
```

### 为什么有效

256-row tile 在小 M 场景下严重浪费 CU：

```
M=256, N=4096, 256x256 tile:
  tiles = ceil(256/256) × (4096/256) = 1 × 16 = 16 tiles
  CU利用率 = 16/304 = 5.3%  ← 极度低下

M=256, N=4096, 128x128 tile:
  tiles = ceil(256/128) × (4096/128) = 2 × 32 = 64 tiles
  CU利用率 = 64/304 = 21.1%  ← 4x 提升
```

### 无需新模板实例化

GFX942 的 `128x128x128_32x32x32_2x2x1` 已通过 `APPLY_CK_GEMM_ALL_LAYOUT_WITH_ARCH`
宏为 RowColQuant / TensorQuant / ABQuantGrouped 全部编译。
GFX950 的 `128x128x128_32x32x64_2x2x1_padding` 也已编译。
仅需在 factory 中添加运行时 M-aware 分支即可。

## 关键结论

1. **N%256 场景**: 256x256→128x128 切换产生 **4x tiles**，加速比 **1.75-3.73x**
2. **N%128 场景**: 256x128→128x128 切换产生 **2x tiles**，加速比 **1.10-1.12x**
3. **大 M 无回退**: M-aware 不触发时行为与基线完全一致 (±0.5%)
4. **LLM 推理场景**: Llama70B gate_proj M=256 加速 **2.68x**，M=512 加速 **1.75x**
